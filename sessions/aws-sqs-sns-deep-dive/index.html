AWS SQS & SNS Deep Dive

AWS SQS SNS Messaging  ·  Intermediate  ·  ~30 min  ·  Updated 2026-06-21

This session covers two AWS services that let the parts of an app talk to each other without waiting on one another:

  • Amazon SQS — a queue that holds work until a worker is ready for it.
  • Amazon SNS — a broadcaster that sends one message to many listeners at once.

We start with a simple analogy, then build up to real patterns, reliability, scaling, and security.

Section 1 · Motivation

Why messaging exists

Most apps start simple: one service calls another and waits for the answer — a synchronous call, like phoning a coworker and staying on the line until they finish. That works until things get busy or one piece breaks:

  • A slow service makes everyone who calls it wait.
  • A crashed service drags down everything that depends on it.
  • A traffic spike on one part forces you to scale all the parts around it.

A quick analogy: ordering coffee ☕

Picture a busy café. If every customer had to stand at the counter until their drink was ready before the next order could be taken, the line would grind to a halt. Instead, you order, get a ticket, and step aside — and the barista works through the tickets at their own pace. A message queue is that ticket system for software: the sender drops off the work and moves on, and a worker picks it up when it's free.

Amazon SQS and SNS bring this idea to AWS — they sit between your services so a sender never waits on a receiver:

  • If a receiver is slow, busy, or briefly down, the message waits safely until it can be handled.
  • The sender (the producer) and receiver (the consumer) run on their own schedules.
  • Either side can fail without taking the other down.

Here's what that buys you in practice:

ConcernTight coupling (direct calls)With SQS / SNS
Downstream slownessCaller blocks and times outProducer enqueues and moves on; consumer drains at its own pace
Downstream crashError propagates upstream immediatelyMessages stay in the queue; consumer retries when it recovers
Burst trafficDownstream gets overwhelmedQueue absorbs the burst; consumer scales independently
Multiple consumersCaller must know about every consumerSNS fans out to all subscribers in one publish call
Audit / replayLost if not explicitly loggedMessages retained in queue up to 14 days

What's the catch?

Once work happens in the background, you give up the simplicity of an instant answer. Messages may arrive slightly out of order, or occasionally be delivered twice — so your code should be able to handle the same message more than once without causing harm. If strict ordering and no duplicates are essential, SQS offers a FIFO mode (covered later) that trades some speed for those guarantees.

Section 2 · SQS

SQS at a glance

Amazon SQS is a pull-based, fully managed message queue — and the oldest service on AWS, so its behaviour is rock-solid and well understood. In short:

  • Producers send messages to a queue URL with the SendMessage API.
  • Consumers poll the queue with ReceiveMessage, process the message, then delete it.
  • AWS handles replication, scaling, and durability behind the scenes.

Key idea: decoupling

SQS exists to decouple applications. Reach for a queue whenever you need to absorb a sudden load spike, avoid timeouts, or make sure no request is lost when a downstream service is busy.

What makes SQS special

♾️ Unlimited throughput & messages

Standard queues have no limit on the number of messages per second you can send, and no limit on how many messages the queue can hold. It scales automatically with your load.

⚡ Low latency

Both publishing and receiving are fast — typically under 10 milliseconds. SQS is built for high-volume, high-velocity workloads.

⏳ Short-lived messages

A message lives in the queue for 4 days by default, up to a maximum of 14 days. It must be received and deleted within that window or it is lost.

🔁 At-least-once delivery

Standard queues guarantee delivery but a message may occasionally arrive twice, and ordering is best-effort. Design consumers to be idempotent. (FIFO queues fix both — see Section 3.)

A handful of per-queue settings define how a queue behaves. The table below lists them; the breakdown that follows explains each one and how to choose a value.

PropertyValue / Limit

Each attribute in detail

Message size — up to 256 KB

A single SQS message (body plus all attributes) can be at most 256 KB. SQS is built for small instructions — "process order 1234", "resize this image" — not for shipping the data itself. If your payload is larger, store the object in S3 and put only the S3 key in the message; the SQS Extended Client library does this for you automatically and transparently handles payloads up to 2 GB. Keeping messages small also keeps cost down, because SQS bills in 64 KB chunks: a 200 KB message counts as four requests, not one.

Retention period — 1 minute to 14 days (default 4 days)

How long a message survives in the queue if nobody successfully processes and deletes it. Once the timer expires, SQS drops the message automatically. The default of 4 days gives you a comfortable window to recover from an outage — if your consumers are down over a long weekend, the work is still waiting when they come back. Set it longer (up to 14 days) for low-volume queues where losing a message is costly; set it shorter for high-volume, time-sensitive data that is worthless if it isn't handled within minutes.

Visibility timeout — 0 s to 12 hours (default 30 s)

When a consumer receives a message, SQS doesn't delete it — it hides it from other consumers for the duration of the visibility timeout. This stops two workers from processing the same message at once. The consumer must finish its work and call DeleteMessage before the timer runs out; if it crashes or is too slow, the message reappears and another consumer picks it up. Rule of thumb: set the visibility timeout slightly higher than your worst-case processing time (e.g. ≥ your Lambda or container task timeout). Too short and you get duplicate processing; too long and a genuinely failed message takes ages to retry.

In-flight messages — 120,000 (Standard) / 20,000 (FIFO)

An "in-flight" message is one that has been received by a consumer but not yet deleted or returned to the queue (i.e. it is inside its visibility timeout). This is a hard quota on concurrency. If you hit the limit, ReceiveMessage returns an error and you must drain the backlog before receiving more. In practice you only approach this ceiling if consumers are far too slow or the visibility timeout is set far too high, so it doubles as a useful signal that something is wrong with throughput.

Delivery guarantee — at-least-once (Standard) / exactly-once (FIFO)

Standard queues guarantee at-least-once delivery: every message arrives, but occasionally a message may be delivered more than once (a side effect of the distributed, highly available design). Your consumers must therefore be idempotent — safe to run twice for the same message, for example by recording a processed message ID and skipping repeats. FIFO queues guarantee exactly-once processing within a deduplication window and preserve strict ordering, at the cost of lower throughput. Pick Standard unless you genuinely need ordering or de-duplication.

Long polling wait — up to 20 s

With short polling (wait = 0), SQS replies immediately even when the queue is empty, so a busy poll loop fires constant empty receives — each one a billable request. Long polling tells SQS to hold the connection open for up to 20 seconds and return as soon as any message arrives. This slashes empty responses, lowers cost, and reduces latency. Almost always set ReceiveMessageWaitTimeSeconds to 20; there is rarely a good reason not to.

Consumers — who reads the queue, and how

A queue is only half the story. A consumer is any process that reads messages, does the work they describe, then deletes them:

  • Producers don't know or care who the consumers are — they just drop messages in.
  • That independence lets you change, scale, or restart consumers without touching producers.
  • Consumers pull messages; AWS never pushes them.

Anything that can call the SQS API can be a consumer:

  • AWS Lambda — the most common choice; Lambda polls the queue for you via an event source mapping.
  • A container or process on ECS / EKS / EC2.
  • Even a plain script running on a server.

The receive → process → delete cycle

Every consumer follows the same three-step loop, and understanding it is the key to using SQS correctly:

  1. Receive — the consumer calls ReceiveMessage and gets one or more messages (up to 10 at a time). At this moment SQS starts the visibility timeout and hides those messages from everyone else.
  2. Process — the consumer does the actual work: charge a card, send an email, resize an image. The message is still in the queue, just invisible.
  3. Delete — only after the work succeeds, the consumer calls DeleteMessage. This is the step beginners forget. If you never delete, the message reappears after the visibility timeout and gets processed again forever.

Delete means "done," not "received"

SQS deliberately does not auto-delete on receive. If a consumer crashes mid-work, the message simply becomes visible again and another consumer retries it — nothing is lost. The flip side: delete only after success, never before, or you'll throw away work that failed.

Competing consumers — scaling out

You can run many identical consumers against one queue — the competing consumers pattern:

  • SQS hands each message to whichever consumer asks next, spreading load automatically.
  • Each message goes to exactly one consumer — work is divided, not duplicated.
  • Add consumers to clear a backlog faster; scale to zero when traffic is quiet.
  • Pair with auto-scaling on queue depth (ApproximateNumberOfMessagesVisible) — see Section 9.

What happens when a consumer keeps failing?

If a message can't be processed — bad data, a downstream bug — the consumer won't delete it, so it returns to the queue and is retried. To stop a "poison" message from looping forever, attach a Dead Letter Queue (DLQ): after a set number of failed attempts (maxReceiveCount), SQS moves the message to the DLQ so the rest of the queue keeps flowing and you can inspect the failure later. Section 7 covers the lifecycle and DLQs in depth.

Section 3 · Queue types

Standard vs FIFO queues

Standard Queue

Best for maximum throughput. Messages may be delivered out of order and more than once — your consumer must be idempotent.

  • Throughput: Nearly unlimited — scales automatically with load.
  • Ordering: Best-effort. AWS tries to preserve order but cannot guarantee it.
  • Delivery: At-least-once. A message may be delivered multiple times if the consumer fails to delete it within the visibility timeout.
  • Use case: Email sending, image resizing, log ingestion, any workload where duplicates are handled by idempotency keys.

Make consumers idempotent

Always design your consumer to tolerate receiving the same message twice. Use a database upsert or a deduplication table keyed on the message's MessageId.

FIFO Queue

Strict ordering and exactly-once processing within a message group. Required when message order matters (financial transactions, state machines).

  • Throughput: Up to 3,000 msg/s with batching; 300 msg/s without. High-throughput FIFO mode: up to 70,000 msg/s.
  • Ordering: Strict FIFO within each MessageGroupId. Different groups process in parallel.
  • Delivery (exactly-once send): Duplicates are removed at the queue level within a 5-minute deduplication window.
  • Name suffix: Queue name must end in .fifo.
  • Use case: Payment processing, order workflows, event sourcing where replay order is critical.

Message Group ID = unit of ordering

Messages with the same MessageGroupId are processed one at a time, in order. Use a user ID, order ID, or any natural partition key. Different groups run in parallel — no head-of-line blocking across groups.

Two ways to deduplicate

1. Deduplication ID: you attach a MessageDeduplicationId to each message. If SQS sees the same ID twice within 5 minutes, the duplicate is discarded.
2. Content-based deduplication: enable this setting and SQS computes a SHA-256 hash of the message body for you — no need to supply an ID. Identical bodies sent within 5 minutes are deduplicated automatically.

Section 4 · SNS

SNS at a glance

Amazon SNS is a push-based, fully managed pub/sub service:

  • A publisher sends one message to an SNS topic.
  • SNS immediately pushes copies to every configured subscriber.
  • No polling required — delivery is push, not pull.

Subscription types

Subscriber typeUse caseDelivery guarantee
SQS queueAsync processing with buffering and retryAt-least-once via SQS
LambdaImmediate serverless invocationAt-least-once; retry on Lambda errors
HTTPS endpointWebhooks to external servicesRetry with backoff; dead-letter after 3 days
Email / Email-JSONHuman alerts and notificationsBest-effort
SMSText alerts and OTPsBest-effort
Kinesis Data FirehoseStream to S3 / Redshift / analyticsAt-least-once via Firehose

Message Filtering

Subscribers can attach a filter policy (JSON) that tells SNS to deliver only messages whose attributes match the policy. This lets a single topic serve multiple consumer types without each consumer receiving every message. Example: an "order-events" topic filters to send status=SHIPPED events only to the logistics queue.

How publishers send to SNS

There are two publishing styles:

  • Topic publish (most common): using the SDK you CreateTopic, register subscribers with Subscribe, then Publish a message to the topic ARN. SNS fans it out to every subscriber.
  • Direct publish (mobile push): used to push to mobile app endpoints. You register the device against a platform application, then publish directly to the endpoint. SNS talks to the platform push service — FCM/GCM (Android), APNS (Apple), and ADM (Amazon).

SNS receives from many AWS services

Many AWS services publish to SNS out of the box, making it a natural notification hub:

  • CloudWatch alarms
  • Auto Scaling lifecycle events
  • Amazon S3 bucket events and RDS events
  • AWS Budgets, Lambda, DynamoDB, and CloudFormation

Scale & limits

A single topic supports up to 12,500,000 subscriptions, and you can create up to 100,000 topics per account — more than enough to fan one event out to a very large audience.

SQS vs SNS — when to use which

The answer is usually: both.

SQS and SNS are complementary. The most common production pattern is SNS → SQS: SNS fans out an event to multiple SQS queues, each owned by a different consumer team. SQS provides the buffer, retry, and DLQ; SNS provides the fan-out. Together they give you resilient, decoupled, observable messaging.

Section 6 · Pattern

The fan-out pattern

Fan-out decouples the event producer from every downstream consumer. When an order is placed, the "Orders" service publishes one event to SNS, and SNS delivers it simultaneously to several independent destinations:

  • an inventory queue
  • a shipping queue
  • an analytics queue
  • an email service

Why SNS → SQS instead of SNS → Lambda directly?

  • Buffer: If the consumer is down or slow, messages wait in the queue (up to 14 days) instead of being lost.
  • Retry: SQS automatically makes the message available again if the consumer crashes mid-processing (visibility timeout expires).
  • DLQ: After N failed attempts, the message moves to a dead-letter queue for inspection — no silent loss.
  • Rate control: Consumers can pull at their own rate, preventing Lambda concurrency spikes or DB overload.

More fan-out patterns

📨 SNS → multiple SQS queues

The core pattern: one publish, many queues, each fully decoupled. Every receiving queue needs an access policy that allows the SNS topic to SendMessage.

🌍 Cross-region delivery

An SNS topic can deliver to SQS queues in other regions, making fan-out a simple building block for cross-region replication and disaster recovery.

🗂️ S3 events → SNS → many queues

S3 can send a given event type to only one destination. To fan a single S3 event out to several consumers, point S3 at an SNS topic and let SNS deliver to many queues.

🔥 SNS → Kinesis Data Firehose → S3

Subscribe a Kinesis Data Firehose stream to the topic to land every message in S3 (or Redshift / OpenSearch) for archival and analytics.

🔢 FIFO fan-out (ordering preserved)

An SNS FIFO topic fanning out to SQS FIFO queues keeps strict ordering and deduplication end to end — use it when every consumer must see events in the exact published order.

🔎 Message filtering

Attach a JSON filter policy so each queue receives only the events it cares about. Example: on an orders topic, the analytics queue subscribes with {"event_type":["order_placed","order_cancelled"]} while the shipping queue takes only {"event_type":["order_placed"]}.

Section 7 · Reliability

Message lifecycle & Dead Letter Queues

Understanding the visibility timeout is essential to building reliable SQS consumers. A message that isn't explicitly deleted becomes visible again after the timeout — enabling automatic retry without any extra retry logic in the producer.

Visibility timeout in depth

When a consumer receives a message, SQS hides it from other consumers for the visibility timeout — 30 seconds by default, configurable from 0 seconds up to 12 hours. If the consumer deletes the message before the timeout expires, it is gone for good. If not (a crash, or a slow run), the message becomes visible again so another consumer can retry it.

  • Set too low → the message reappears while the first consumer is still working on it, so it gets processed twice.
  • Set too high → if a consumer dies, the message stays hidden for a long time before anyone can retry it, adding latency.
  • Rule of thumb: set it a little longer than your worst-case processing time.

Need more time? Call ChangeMessageVisibility

A consumer that needs longer than expected can call the ChangeMessageVisibility API to extend the timeout for that specific in-flight message — buying more time without letting the message become visible to others.

Dead Letter Queues (DLQ)

A DLQ is a separate SQS queue that receives messages that fail to be processed after maxReceiveCount attempts. It is your safety net for poison-pill messages — messages that consistently cause consumer errors.

  • Set maxReceiveCount = 3–5 for most workloads. Set higher if temporary errors (network blips) are common.
  • DLQ retention: set longer than the source queue. You want time to inspect failures.
  • Monitor ApproximateNumberOfMessagesVisible on the DLQ and alarm when it is > 0.
  • SNS subscriptions also support a DLQ for failed delivery attempts — set one on every production subscription.

A DLQ without an alarm is worthless

Messages silently pile up in the DLQ while your system appears healthy. Always create a CloudWatch alarm on ApproximateNumberOfMessagesVisible for every DLQ. Without it you may not notice a processing failure for days.

Section 8 · Efficiency

Long polling vs short polling

When a consumer calls ReceiveMessage, SQS has to decide how hard to look for a message before answering. That choice — short polling vs long polling — has a big impact on cost and latency.

Short polling (default)

SQS samples only a subset of its servers and replies immediately — even if that means returning an empty response when messages actually exist elsewhere.

  • Replies instantly, empty or not
  • More empty receives → more API calls → higher cost
  • WaitTimeSeconds = 0

Long polling (recommended)

SQS waits up to WaitTimeSeconds for a message to arrive before replying, and queries all servers — so it never misses a waiting message.

  • Eliminates empty responses → fewer API calls, lower cost
  • Lower latency: the message is returned the moment it arrives
  • WaitTimeSeconds between 1 and 20 seconds

How to enable it

Turn on long polling at the queue level with the ReceiveMessageWaitTimeSeconds attribute (applies to every receive), or per request at the API level by passing WaitTimeSeconds on ReceiveMessage (this overrides the queue setting). AWS recommends 20 seconds for almost all workloads.

Section 9 · Scaling

SQS & Auto Scaling

Because a queue absorbs bursts, you can size your consumer fleet to the work waiting in the queue rather than to the raw incoming request rate. This is one of the most common SQS patterns on AWS.

Scale on queue depth, not CPU

Publish messages to the queue, then put a CloudWatch alarm on the ApproximateNumberOfMessagesVisible metric (the queue length). When the backlog grows, the alarm tells an Auto Scaling Group to add consumer instances; when the queue drains, it scales back in. Consumers can be EC2 instances, containers (ECS/EKS), or Lambda functions.

Decouple application tiers

A classic example is media processing: a lightweight front tier accepts uploads and drops a job on the queue; a heavier back tier of GPU instances pulls jobs and processes the video. Each tier scales on its own — a spike in uploads simply lengthens the queue instead of overwhelming the workers.

Buffer database writes

Put SQS in front of a database that cannot absorb sudden write spikes. The application writes to the queue at any rate; consumers drain it at a steady, database-friendly pace. The queue smooths the load and no writes are lost during a spike.

Section 10 · Security

Security

Access control (IAM + resource policies)

SQS and SNS have two policy layers:

  • Identity-based (IAM): Attach policies to IAM roles/users. Best for same-account access.
  • Resource-based (queue/topic policy): Required for cross-account access, and for allowing SNS to send to SQS. Always scope to specific principals and actions — avoid * principals.

Encryption

  • In transit: All API calls use TLS. Use HTTPS endpoints. Enforce with a condition aws:SecureTransport: true in the queue policy.
  • At rest (SSE-SQS): AWS-managed key. Zero config, no extra cost. Sufficient for most workloads.
  • At rest (SSE-KMS): Customer-managed key. Required for regulatory environments. Adds per-request KMS charges; consider batching.

VPC endpoints (PrivateLink)

Create an interface VPC endpoint for SQS and SNS. Traffic from your VPC stays on the AWS backbone — it never touches the public internet. Combine with an endpoint policy to restrict which queues/topics can be accessed.

Least-privilege checklist

  • Producers: sqs:SendMessage only
  • Consumers: sqs:ReceiveMessage, sqs:DeleteMessage, sqs:GetQueueAttributes only
  • SNS publish: sns:Publish scoped to specific topic ARN
  • SQS queue policy: allow SNS service principal to SendMessage
  • Never use * as principal in a resource policy on production queues
Section 11 · Observability

What to monitor

Five CloudWatch metrics cover 90% of "is my messaging healthy?" — alarm on these first.

SQS

ApproximateNumberOfMessagesVisible

Queue depth — messages waiting to be processed. Rising depth = consumers are falling behind. Alarm at your SLA-based threshold.

SQS

ApproximateAgeOfOldestMessage

How old is the oldest unprocessed message? A rising age means messages are stuck. Alarm well before your retention window expires.

SQS DLQ

ApproximateNumberOfMessagesVisible (DLQ)

Any message here means something failed > maxReceiveCount times. Alarm at > 0. Investigate immediately.

SNS

NumberOfNotificationsFailed

SNS failed to deliver to a subscriber. Check endpoint health, queue permissions, and subscription filter policies.

SNS

NumberOfMessagesPublished

Sudden drop may mean the upstream service stopped publishing. Correlate with your application deploy events.

SQS

NumberOfMessagesSent vs NumberOfMessagesDeleted

If sent >> deleted over time, consumers are not keeping up. Scale out or debug consumer throughput.

Section 12 · Production

A production reference architecture

A safe default for event-driven workloads: one SNS topic fans out to multiple SQS queues; each queue has a DLQ; encryption at rest and in transit; VPC endpoints; and CloudWatch alarms on depth and age.

Six-point production checklist

  • Every SQS queue has a DLQ with maxReceiveCount set to 3–5.
  • CloudWatch alarm on DLQ depth > 0 and on source queue age exceeding your SLA.
  • Encryption at rest: SSE-KMS for regulated data, SSE-SQS otherwise.
  • Queue policy enforces aws:SecureTransport: true — no plaintext HTTP.
  • VPC interface endpoints for SQS and SNS — no traffic over public internet.
  • IAM least privilege: separate roles for producers and consumers.
Section 13 · One slide to remember

The four-card mental model

If you remember nothing else from this session, remember these four roles. They answer the most common architecture questions.

Section 14 · Pitfalls

Common mistakes
"We'll set visibility timeout to 30 s because that's the default."

If your consumer takes 60 seconds to process, the message becomes visible again after 30 s — while it's still being processed. Another consumer picks it up, causing a duplicate. Set visibility_timeout_seconds to at least 6× your average processing time, or extend it dynamically using ChangeMessageVisibility.

"We don't need a DLQ — our consumer handles errors."

A poison-pill message (bad JSON, unexpected schema, downstream dependency that's permanently broken) will loop forever, exhausting your consumer's threads and blocking other messages. Without a DLQ it retries indefinitely until the 14-day retention expires. Always configure a DLQ with maxReceiveCount.

"We publish directly to all 10 SQS queues from the service."

The producer must now know about every consumer — tight coupling through the back door. Adding a new consumer requires a code change in the producer. Use SNS as the single publish target; SNS fans out to all queues. Adding a consumer is a new SNS subscription, invisible to the producer.

"FIFO is safer — let's use it everywhere."

FIFO throughput is capped (300–70,000 msg/s vs nearly unlimited for Standard). For high-volume workloads where order doesn't matter (image resize, email send), FIFO is an unnecessary bottleneck. Use FIFO only when ordering or exactly-once matters. Idempotent Standard queues handle the vast majority of use cases.

"We use short polling because long polling sounds more complex."

Short polling returns immediately even when the queue is empty, incurring cost per empty response. Long polling (up to 20 s) waits for a message before returning — dramatically reducing empty receives and API costs. Set receive_wait_time_seconds = 20 on every queue. There is no meaningful complexity increase.

"We store sensitive data (PII, tokens) directly in message bodies."

SQS message bodies appear in CloudWatch Logs, X-Ray traces, and potentially in DLQ payloads inspected by engineers. Store sensitive data in S3 (with server-side encryption) and put only a reference (S3 key) in the message body. This is the "claim-check" pattern and also helps with the 256 KB message size limit.
